# OmniStudio Field Validation Suite

A reusable Apex field-validation library (email, US phone, US ZIP code) exposed to
Salesforce OmniStudio as Integration Procedures, wired into a working two-step
OmniScript (`ContactUsDemo`) that validates each field in real time and shows the
result with a visual checkmark/X — plus a resolved city/state for valid ZIP codes.

This is a **complex, multi-layer OmniStudio component** — not a single class or
config item. It spans four distinct architectural layers (Apex services → a
`System.Callable` bridge → Integration Procedures → an OmniScript UI), each with
its own real integration challenges, several of which required deliberate
debugging rather than working correctly on the first attempt.

## Built with Vibe Coding — and where that stopped being possible

The Apex layer (all four classes below, plus their test classes) was built using
an AI-assisted ("Vibe Coding") workflow with Claude AI in VS Code — every class
was generated, then manually reviewed, stress-tested with deliberate edge cases,
and fixed where real bugs were found, before being trusted.

**The OmniStudio layer — the Integration Procedures and the `ContactUsDemo`
OmniScript — was built manually, by hand, in the OmniStudio Designer UI.** This
wasn't a stylistic choice: this org's OmniStudio version (240.5.2.1, Winter 2023)
stores Integration Procedures and OmniScripts as `OmniProcess`/`OmniProcessElement`
records that depend on a compiled cache (`OmniProcessCompilation`) which is only
generated by the Designer UI's own save/activate flow. Direct Apex DML into these
objects — and even OmniStudio's own DataPack export/import framework — were both
investigated and confirmed **not** to reliably produce a working, compiled result
in this org's version. So the declarative layer had to be built the way any real
OmniStudio developer builds it: by hand, in the Designer.

## Apex classes (`force-app/main/default/classes`)

| Class | Purpose |
|---|---|
| `EmailValidationService` | `isValidEmail()` / `normalizeEmail()` — regex-based email format validation. |
| `PhoneFormatService` | `isValidUSPhone()` / `formatUSPhone()` — validates and formats US phone numbers, including NANP structural checks. |
| `ZipCodeLookupService` | `lookupZipCode()` — validates a US ZIP (including ZIP+4) and looks up city/state via a live HTTP callout to [Zippopotam.us](https://api.zippopotam.us). |
| `FieldValidationIPBridge` | Bridges the three services above to OmniStudio by implementing `System.Callable`, so they can be invoked as a Remote Action from an Integration Procedure. |

Each service class has a matching `*Test` class with 100% code coverage.

## Real bugs found and fixed along the way

This project's development process deliberately included adversarial edge-case
testing at every layer, not just happy-path checks. A few of the real issues found
and fixed:

- **Email**: leading dots and consecutive dots (`.test@x.com`, `test@x..com`) were
  originally accepted as valid — fixed with a tightened regex.
- **Phone**: an all-zeros number (`0000000000`) originally passed as "valid" purely
  because it had the right digit count — fixed by adding NANP structural checks
  (area code / exchange code can't start with 0 or 1).
- **ZIP code**: whitespace-padded input and the common ZIP+4 format
  (`94103-1234`) were originally rejected outright — both fixed to handle real,
  common user input.
- **Integration Procedure layer**: all three Integration Procedure Actions
  initially wrote their responses to the same flat root-level JSON object, and
  since all three share a field named `isValid`, each action's result silently
  overwrote the previous one's — causing the OmniScript's validation indicators to
  never resolve correctly. Fixed by setting a distinct `responseJSONNode` on each
  action so responses nest under their own key instead of colliding.
- **Bridge class**: a non-string input (e.g., a misconfigured Number-type field
  sending a JSON number instead of text) caused a raw type-cast exception that
  produced an inconsistent response shape — fixed with a safe string conversion
  and a consistent error contract.

## A known, investigated limitation

The OmniScript currently validates each field on a second step (after clicking
Next), rather than live as the user types. Live field-level triggering (firing an
Integration Procedure automatically on blur/change) was investigated thoroughly —
searching all 476 elements across this org for any real precedent, and
cross-referencing official Salesforce documentation — and confirmed to have no
built-in, config-level mechanism in this OmniStudio version. The only documented
alternative (a custom Lightning Web Component override) is a materially larger
scope of work, not a simple toggle, and was deliberately left as a known
enhancement rather than pursued without a clear, confirmed path.

## OmniStudio Integration Procedures

Three Integration Procedures wrap the bridge for use from any OmniScript:

- `ValidateEmailIP_Default` — input `emailInput` → output `isValid`, `normalizedEmail`
- `ValidatePhoneIP_Default` — input `phoneInput` → output `isValid`, `formattedPhone`
- `ValidateZipCodeIP_Default` — input `zipInput` → output `isValid`, `city`, `state`, `stateAbbreviation`

Invoked via:

```apex
omnistudio.IntegrationProcedureService.runIntegrationService(
    'ValidateEmailIP_Default',
    new Map<String, Object>{ 'emailInput' => 'test@example.com' },
    new Map<String, Object>()
);
```

## Repository contents

This repo tracks the Apex layer only. The Integration Procedures and the
`ContactUsDemo` OmniScript are OmniStudio org configuration (`OmniProcess` /
`OmniProcessElement` records), not deployable source files, and are built directly
in the org rather than included here.
